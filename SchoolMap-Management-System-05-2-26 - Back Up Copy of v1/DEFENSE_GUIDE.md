## TEST 3: ROLLBACK DEMO - Simulate Failure & Recovery

### Professor's Request:
> "Simulate a system crash during a transaction. Show me that the data successfully rolls back."

### Your Response (Click-by-Click):

#### Step 1: Connect as Root
```bash
# Exit app user session
EXIT;

# Reconnect as root
mysql -u root -p school_map_db
```

#### Step 2: Show Initial State
```sql
-- Count current visitor logs
SELECT COUNT(*) AS visitor_count FROM visitor_logs;
```

**Sample Output:**
```
| visitor_count |
|---|
| 1 |
```

**Say to Professor:**
> "We currently have 1 visitor log in the database."

#### Step 3: Start Transaction & Add Multiple Records
```sql
SET @current_admin_id = 1;

START TRANSACTION;

INSERT INTO visitor_logs (name, purpose, destination, category, time_in, date)
VALUES ('Visitor A', 'Meeting', 'Office A', 'official', '10:00:00', CURDATE());

INSERT INTO visitor_logs (name, purpose, destination, category, time_in, date)
VALUES ('Visitor B', 'Document', 'Office B', 'academic', '11:00:00', CURDATE());

INSERT INTO visitor_logs (name, purpose, destination, category, time_in, date)
VALUES ('Visitor C', 'Service', 'Maintenance', 'service', '12:00:00', CURDATE());

-- At this point, transaction is open but NOT committed
```

#### Step 4: Check Count Inside Transaction
```sql
-- Inside transaction, should see 4 records (original 1 + 3 new)
SELECT COUNT(*) AS visitor_count FROM visitor_logs;
```

**Expected Output:**
```
| visitor_count |
|---|
| 4 |
```

**Say to Professor:**
> "Inside the transaction, we see 4 visitor logs. But since we haven't committed, these changes are isolated and not permanent."

#### Step 5: Simulate Crash - ROLLBACK
```sql
-- Simulate system crash by rolling back
ROLLBACK;
```

#### Step 6: Verify Rollback
```sql
-- After ROLLBACK, count should return to original
SELECT COUNT(*) AS visitor_count FROM visitor_logs;
```

**Expected Output:**
```
| visitor_count |
|---|
| 1 |
```

**Say to Professor:**
> "Perfect! After the ROLLBACK (simulating a crash), we're back to 1 visitor log. All 3 temporary records were discarded atomically. The database is consistent. This demonstrates the **ATOMICITY** principle of ACID — either ALL operations succeed and commit, or NONE do."

#### Step 7: Demonstrate Successful Commit
```sql
-- Now do a transaction that COMMITS successfully
SET @current_admin_id = 1;

START TRANSACTION;

INSERT INTO visitor_logs (name, purpose, destination, category, time_in, date)
VALUES ('Visitor D', 'Demo', 'Room 101', 'official', '14:00:00', CURDATE());

COMMIT;

-- Verify it persisted
SELECT COUNT(*) AS visitor_count FROM visitor_logs;
```

**Expected Output:**
```
| visitor_count |
|---|
| 2 |
```

**Say to Professor:**
> "When we COMMIT successfully, the record is permanent. We now have 2 visitor logs, and this persists even if the system crashes later."

## ADDITIONAL QUESTIONS PROFESSOR MIGHT ASK

### Q1: "How do you prevent an admin from being deleted if they have audit log entries?"

**Your Answer:**
```sql
-- Show the foreign key constraint
SELECT CONSTRAINT_NAME, REFERENCED_TABLE_NAME, COLUMN_NAME
FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
WHERE TABLE_NAME = 'audit_log' AND COLUMN_NAME = 'user_id';
```

**Response:**
> "The audit_log table has a FOREIGN KEY reference to admin.user_id. If we try to delete an admin who has audit entries, MySQL will reject it automatically:
> ```
> ERROR 1451 (23000): Cannot delete or update a parent row: a foreign key constraint fails
> ```
> This maintains **referential integrity**."

---

### Q2: "How do you ensure that the database remains consistent?"

**Your Answer:**
```sql
-- Show the transaction example
START TRANSACTION;
-- Make changes here
-- If any statement fails, automatic ROLLBACK
COMMIT;
```

**Response:**
> "I use ACID transactions for critical operations. Either ALL statements succeed and COMMIT, or NONE do (automatic ROLLBACK). This ensures the database is never in an inconsistent state. Additionally, all foreign key constraints and data validation rules in stored procedures enforce consistency."

---

### Q3: "What happens if two admins try to update the same record simultaneously?"

**Your Answer:**
```sql
-- Show row-level locking
SET innodb_lock_wait_timeout = 50;
START TRANSACTION;
SELECT * FROM pins WHERE pin_id = 1 FOR UPDATE;  -- Acquire lock
UPDATE pins SET name = 'Updated' WHERE pin_id = 1;
COMMIT;  -- Release lock
```

**Response:**
> "MySQL InnoDB uses **row-level locking**. When Admin A updates a pin, that specific row is locked. If Admin B tries to update the same row, B waits up to 50 seconds for A's lock to release. After A commits, B gets the lock and makes their update. This prevents conflicting updates and maintains data integrity. If the wait time exceeds 50 seconds, a deadlock error is returned and the transaction restarts."
